# Data Simulation Documentation

This document provides a technical overview of how the `scripts/data_simulation.py` module generates the synthetic dataset for the Manila Transit Analytics project.

## 1. Overview
The goal of the simulation is to create a realistic but "dirty" dataset that reflects the intermodal commuting challenges in Metro Manila. The script generates over 700 rows of data across 7 attributes, incorporating geographical accuracy and statistical noise.

## 2. Core Logic

### Geographic Accuracy
The script uses a mapping of actual transit systems to their respective stations:
- **EDSA Carousel**: 26 stations (Monumento to PITX).
- **MRT-3**: 13 stations (North Ave to Taft Ave).
- **LRT-2**: 13 stations (Recto to Antipolo).
- **LRT-1**: 25 stations (Fernando Poe Jr. to Dr. Santos).

Each row is generated by first selecting a **Transit Line** and then randomly selecting a valid **Station** from that specific line's route.

### Wait Time Calculation
The `Actual_Wait_Time` is not purely random; it is calculated using a base scheduled interval modified by environmental factors:

1. **Base Interval**: A random integer between 5 and 10 minutes.
2. **Time of Day Multiplier**:
   - **Peak Morning/Evening**: 2.5x multiplier (simulating heavy congestion).
   - **Late Night**: 0.8x multiplier (simulating fewer passengers/faster turnaround).
   - **Mid-Day**: 1.0x multiplier.
3. **Weather Impact**:
   - Uses a **Gamma Distribution** (`np.random.gamma`) to simulate skewed delay patterns.
   - **Heavy Rain**: Adds significant delay (mean ~15 mins for EDSA Carousel, mean ~4.5 mins for trains).
   - **Clear/Cloudy**: Adds minor variance (mean ~2.25 mins).

## 3. "Dirty Data" Injection
To satisfy the grading requirements for **Data Cleaning and Preprocessing**, the script programmatically injects errors using a configurable constants block (`CONFIG`).

### Configuration Block
The following parameters can be adjusted at the top of the script:
- `missing_ratio`: Percentage of rows with missing wait times.
- `inconsistent_ratio`: Percentage of rows with categorical string errors.
- `outlier_count`: Total number of Gaussian noise records.
- `duplicate_count`: Number of identical rows to append.

### A. Missing Values (Nulls)
Records are replaced with `NaN` (Not a Number) to simulate sensor failure or data logging errors.

### B. Inconsistent Categorical Data
Unlike the initial version, string inconsistencies are now applied across **all categorical columns** (`Transit_Line`, `Station_From`, `Time_of_Day`, `Weather_Condition`). A specialized mutation function randomly applies:
- **Case Variations**: Lowercase, uppercase, or swapped case (e.g., `MRT-3` -> `mrt-3`, `cUBAO`).
- **Whitespace Injection**: Adding leading/trailing spaces (e.g., `  Clear `).
- **Character Removal**: Removing hyphens or internal spaces (e.g., `MRT 3`, `NorthAve`).

### C. Gaussian Outliers (Statistical Noise)
Instead of hardcoded "magic numbers," the script uses **Gaussian (Normal) Distributions** to create two types of unreliable records:
- **Impossible Negatives**: Generated using $N(\mu=-15, \sigma=3)$. These represent system underflow errors.
- **Extreme Delays**: Generated using $N(\mu=500, \sigma=75)$. These represent extreme system glitches or "ghost" data.

### D. Duplicate Records
The script samples **10 identical rows** and appends them back into the dataset to test duplicate detection logic.

## 4. Usage
To regenerate the raw dataset, run the following command from the project root:
```powershell
python scripts/data_simulation.py
```
The output will be saved to `data/1_raw/manila_transit_raw.csv`.
